# 📜 游戏策划与开发文档 (GDD)：AI 驱动的终端诡宅冒险

---

## 1. 项目愿景与基调

**项目目标：** 在极限开发周期内，完成一款运行在终端（Terminal）的克苏鲁风格文字冒险游戏。

**核心基调：** 惊悚、未知、动态叙事。采用极简的 ASCII UI 结合大语言模型（LLM）的强大能力，实现无限探索的动态地图、随机遭遇以及突发的结局翻转。

**技术栈：** Python 3.11+ ＋ Ollama 本地模型

## 2. 世界观与叙事设定

### 2.1 背景故事

时间：1923年，新英格兰沿海小镇 **阿卡姆（Arkham）** 郊外。

玩家扮演一名接到匿名信的**调查者**，信中提及镇郊的"灰石庄园"数十年前曾发生过一起集体失踪事件，而近日有牧羊人声称再次看见庄园中透出幽光。带着对真相的渴望（或贪婪），你独自推开了那扇腐朽的铁门。

庄园深处封印着某种**远古存在**的碎片意识——它不会直接杀死你，而是慢慢侵蚀你的理智，直到你主动为它打开封印，或是带着破碎的心智永远留在这里。

### 2.2 角色背景（三选一，影响初始属性）

开局玩家从以下三个背景中选择，影响初始属性与部分AI叙事风格：

| 背景 | 描述 | HP | SAN | 特殊能力 |
|------|------|----|-----|----------|
| **记者** | 习惯挖掘真相，擅长收集线索 | 8 | 7 | 每次进入新房间额外获得一条"碎片线索"，有助于触发更高质量的AI叙事 |
| **医生** | 受过严格理性训练，心理较为稳健 | 7 | 9 | SAN归零阈值更高，首次SAN归零时有一次"理性抵抗"豁免 |
| **盗墓者** | 见过太多诡异之事，身手敏捷 | 10 | 5 | 战斗检定时附加 +1 修正，但初始SAN极低，更快触发异变 |

### 2.3 开场叙事（固定文本）

> 铁门在你身后轰然关闭。  
> 黑暗吞没了来时的路。  
> 空气中有一股腐烂的甜味——  
> 像是某种东西正在等待你。  
>  
> 你握紧手中的手电筒，向前走去。

## 3. 核心玩法循环 (Core Loop)

游戏分为两个阶段：**探索期** 与 **异变期**。

### 3.1 探索期 (The Exploration)

- 玩家在未知的 2D 坐标系地图中移动（初始坐标为 `0,0`）。
- 探索未知区域时，系统实时调用 AI 生成房间描述，并进行遭遇判定（普通事件 / 收集物品 / 发现受诅物）。
- 玩家根据文字描述输入决策，系统通过 AI 解析逻辑并判定结果，动态更新玩家的 HP、SAN 或物品栏。

### 3.2 异变期 (The Anomaly / 终局)

- 当玩家获得"受诅物"时，触发**诅咒检定**（见第6节）。随着受诅物增多，触发"异变"的概率上升。
- 检定失败或探索房间数达到阈值（20间），游戏强制切换至"异变模式"。
- AI 完全接管战局充当 DM，根据玩家当前的物品储备和所处环境，即时生成专属的终极危机（Boss战或逃生任务），进入连续判定的生死存亡阶段。

## 4. 多结局分支系统

### 4.1 设计原则

结局由 AI 根据玩家的**最终状态快照**动态生成叙事文本，保证每局结局唯一。程序只负责判断"走哪条结局路径"，具体文字由AI生成。

### 4.2 结局触发条件

| 结局类型 | 触发条件 | 基调 |
|----------|----------|------|
| **逃脱·理智尚存** | HP > 0，SAN > 3，找到"逃脱出口"物品 | 惊魂后的生还，带走了部分真相 |
| **逃脱·精神破碎** | HP > 0，SAN ≤ 3，找到"逃脱出口"物品 | 肉体逃离，但理智已被侵蚀，结局充满不祥预兆 |
| **融合·成为祭品** | SAN = 0，HP > 0 | 玩家主动为远古存在打开封印，获得"永恒"但失去自我 |
| **战死·无名英雄** | HP = 0，诅咒值 < 3 | 死于探索中，庄园仍封存着秘密 |
| **湮灭·彻底消失** | HP = 0，诅咒值 ≥ 3，或异变期战败 | 最坏结局，连尸骨都不会被找到 |

### 4.3 AI 结局生成 Prompt 框架

触发结局时，将以下上下文注入给 AI：
- 角色背景、最终 HP/SAN 值
- 持有的物品列表
- 探索过的房间数量与关键事件记录
- 结局类型标识

AI 据此生成 200-400 字的专属结局叙事，保证复玩差异性。

## 5. 工程实现策略

为了确保快速跑通核心流程，工程架构遵循"数据与表现分离"及"接口标准化"原则。

### 5.1 核心数据结构 (State Management)

**玩家状态 (Player State)：**
```python
{
  "name": str,
  "background": str,          # journalist / doctor / looter
  "hp": int,                  # 当前体力，初始值依背景而定
  "hp_max": int,
  "san": int,                 # 当前理智值
  "san_max": int,
  "inventory": list[str],     # 物品名称列表
  "curse_level": int,         # 受诅物数量，触发异变的关键变量
  "rooms_visited": int,       # 已探索房间数
  "key_events": list[str]     # 关键事件记录，用于结局生成
}
```

**地图存储 (Map Grid)：** 坐标哈希表，键为 `(x, y)` 元组，值为房间数据对象。

```python
{
  (0, 0): {
    "name": "入口大厅",
    "description": "...",   # AI生成
    "visited": True,
    "exits": ["north", "east"],
    "event_triggered": True
  }
}
```

优势：支持地图无限边界与按需生成，规避复杂的寻路算法与数组越界处理。

### 5.2 AI 数据交互标准 (JSON Output)

AI 响应必须通过 Prompt 约束强制输出 JSON 格式，供主程序解析并修改内存状态。

**标准 API 响应格式：**
```json
{
  "narration": "画像中的人突然发出一声尖叫！你的精神受到了剧烈冲击。",
  "hp_delta": 0,
  "san_delta": -2,
  "new_item": null,
  "is_cursed": false,
  "options": [
    {"key": "1", "text": "仔细检查画像"},
    {"key": "2", "text": "转身离开这个房间"},
    {"key": "3", "text": "用手帕遮住眼睛继续前进"}
  ]
}
```

**容错处理：** 若 AI 返回 JSON 解析异常，启用 Fallback 默认事件，防止主线程崩溃。

## 6. AI 接口技术规格

### 6.1 模型选择

使用 **Ollama** 本地部署，推荐模型优先级：

| 优先级 | 模型 | 理由 |
|--------|------|------|
| 首选 | `llama3.1:8b` | 叙事质量与速度平衡，本地运行流畅 |
| 备选 | `mistral:7b` | 指令遵循能力强，JSON输出更稳定 |
| 备选 | `gemma2:9b` | 中文支持较好（如需双语） |

### 6.2 调用方式

使用 Ollama Python SDK（`ollama` 包）或直接调用 HTTP API：

```python
import ollama

response = ollama.chat(
    model="llama3.1:8b",
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": user_context}
    ],
    format="json"   # 强制JSON输出
)
```

**接口地址：** `http://localhost:11434`（Ollama 默认端口）

### 6.3 Context 长度控制

- System Prompt 控制在 500 tokens 以内
- 每次请求只传入**当前房间上下文 + 玩家状态摘要**，不传完整历史
- 关键事件用 `key_events` 列表摘要保存，避免 context 膨胀

## 7. 核心功能模块划分 (Features & Modules)

### 模块一：终端 UI 与输入控制器 (CLI Controller)

**功能描述：** 游戏的"前端显示"。负责终端界面的渲染、状态刷新及捕获玩家输入指令。

**工程细节：**
- **主控循环 (Main Loop)：** 持续监听用户输入状态
- **UI 刷新：** 每次有效操作后执行清屏（`os.system('clear')`）
- **HUD 抬头显示：** 在屏幕顶部固定渲染玩家 HP/SAN 及物品栏
- **视野渲染：** 读取地图字典中当前坐标周围 8 个格子的状态，用 ASCII 字符拼装出 3×3 的迷你小地图
- **指令路由：** 拦截 `w/a/s/d`（移动）和数字键（选项），触发后续模块逻辑

### 模块二：状态机与地图引擎 (Map Engine)

**功能描述：** 游戏的核心业务逻辑，处理坐标计算与状态流转。

**工程细节：**
- 拦截模块一传来的移动指令，计算目标坐标 `(x', y')`
- 检查坐标哈希表，若坐标无数据，向模块三发起请求，获取新房间数据并写入字典
- 封装基础的状态修改方法（如 `apply_damage(amount)`、`apply_san_loss(amount)`）

### 模块三：AI 管道与遭遇生成器 (AI Pipeline)

**功能描述：** 游戏的"剧情引擎"。负责构建 Prompt 上下文并对接 Ollama API。

**工程细节：**
- **房间生成器 (Room Generator)：** 传入坐标、周边房间信息、玩家状态，返回结构化的新房间叙事 JSON
- **意图解析器 (Intent Parser)：** 接收玩家选择的行动选项编号，结合当前环境上下文发送给 AI，返回数值变动 JSON
- **结局生成器 (Ending Generator)：** 接收玩家最终状态快照与结局类型，生成专属结局文本

### 模块四：战斗与检定系统 (Check System)

**功能描述：** 处理所有需要随机判定的游戏事件。

**工程细节：**

**属性检定（隐藏掷骰）：**
- 玩家选择行动后，后台执行 `random.randint(1, 20)` 掷 d20
- 将骰子结果、玩家相关属性、行动描述一并发送给 AI
- AI 根据数值大小裁定成败并生成叙事，玩家只看到文字结果，不暴露数字

```python
def skill_check(stat_value: int, difficulty: int = 10) -> tuple[bool, int]:
    roll = random.randint(1, 20)
    # 背景加成已在此处应用
    modified = roll + (stat_value // 3)
    return modified >= difficulty, roll
```

**战斗流程：**
1. AI 生成敌人描述与3个行动选项（稳妥/冒险/鲁莽）
2. 玩家选择行动
3. 后台执行检定，将结果传给 AI
4. AI 生成战斗叙事并返回 `hp_delta`/`san_delta`

### 模块五：异变检定与终局系统 (Climax System)

**功能描述：** 游戏的高潮翻转与最终结局判定。

**工程细节：**

**诅咒检定公式（MVP版本）：**
```python
import random

def curse_check(curse_level: int) -> bool:
    """返回 True 表示触发异变"""
    threshold = min(curse_level * curse_level * 5, 80)  # 概率上限80%
    return random.randint(1, 100) <= threshold
```

| 受诅物数量 | 触发概率 |
|-----------|----------|
| 1 | 5% |
| 2 | 20% |
| 3 | 45% |
| 4 | 80%（上限）|

**终局接管：** 触发异变后，挂起地图探索功能，将玩家完整状态注入 System Prompt，进入文本交互生存模式，直至满足某一结局触发条件。

## 8. 极限开发协作规范 (Game Jam Rules)

### 人员分工

| 角色 | 负责模块 |
|------|----------|
| 开发者 A | 模块一（CLI UI）+ 模块二（地图引擎）|
| 开发者 B | 模块三（AI Pipeline）+ 模块四/五（检定与终局）|
| AI 辅助 | 样板代码生成、Prompt 调优、Debug 协助 |

### 并行开发策略

- 建立 GitHub 仓库，各自拉 feature 分支：`feature/ui-map`、`feature/ai-pipeline`
- **Mock 数据先行：** 开发者 A 前期使用 `mock_ai_response()` 返回静态 JSON，优先跑通终端界面和主循环，不阻塞等待真实 AI 接口
- 接口契约以第 5.2 节 JSON 格式为准，双方共同遵守，不得私自改动字段

### 版本控制规范

- `main` 分支只合并可运行的代码
- 每完成一个模块的 MVP 功能即发起 PR
